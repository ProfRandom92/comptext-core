# Specification: CompText Core Improvements (spec.md)

This specification defines the functional improvements to the cryptographic verification chain and the LLM Provider Router in `comptext-core`.

---

## 1. Cryptographic Verification Chain Optimization

### Objective
Minimize the token overhead when passing Merkle proofs to LLM contexts or storing them in space-constrained trace ledgers.

### Problem
Merkle proofs generated by `MerkleTree.try_generate_proof` return a JSON payload:
```json
{
  "leaf_hash": "hex_string_64_chars",
  "proof_path": [
    "hex_string_64_chars",
    "hex_string_64_chars"
  ],
  "root_hash": "hex_string_64_chars"
}
```
Each 32-byte hash is represented as a 64-character hex string. In LLM prompts:
1. Hex strings are heavily fragmented by tokenizers (typically 8–16 tokens per hash).
2. The JSON structure adds syntax characters (brackets, quotes, keys).
3. Redundant duplicates (e.g., duplicated nodes when a level has an odd length) are stored in the path.

### Requirements
- **Base64 Representation**: Support serialization of 32-byte hashes using URL-safe Base64 without padding (which yields 43 characters instead of 64 hex characters, saving ~33% characters and reducing token counts).
- **Redundancy Elimination**: During tree traversal, if a node's sibling is itself (due to odd level length), omit it or store a 1-character placeholder (e.g., `""` or `"*"`), letting the verifier compute the duplicate node value without needing the duplicate hash in the proof array.
- **Compact Format**: Define a compact string representation of the proof:
  `[leaf_index_or_hash_b64]|[proof_hash_1_b64],[proof_hash_2_b64],...`

## 2. LLM Provider Router Exception Handling

### Objective
Unify exception handling across all supported LLM providers (`GoogleProvider`, `XAIProvider`, `NVIDIAProvider`) to shield consumers from SDK-specific or protocol-specific exceptions.

### Problem
Currently:
1. `XAIProvider` raises raw `httpx.HTTPStatusError` / `httpx.HTTPError`.
2. `GoogleProvider` raises raw `google.genai` SDK exceptions.
3. `NVIDIAProvider` retries on `RateLimitError` but raises a raw `RuntimeError` if exhausted; other errors bubble up as raw `openai.OpenAIError` exceptions.
A caller of `get_provider` has to write provider-specific `except` blocks.

### Requirements
- **Standardized Exceptions**: Define a common base exception class `ProviderError` and subclasses:
  - `ProviderAuthError`: Missing/invalid API keys or credentials.
  - `ProviderRateLimitError`: Rate limit exceeded (HTTP 429).
  - `ProviderConnectionError`: Network timeout, connection failure, or unreachable host.
  - `ProviderAPIError`: Standard API errors (invalid payload, server errors, HTTP status errors).
- **Consistent Wrapper**: Wrap all provider API calls so that they catch SDK/library exceptions and raise the corresponding standardized `ProviderError`.
- **Pre-flight Key Checks**: Check for missing keys during initialization and raise `ProviderAuthError` immediately.
